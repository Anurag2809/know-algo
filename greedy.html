Greedy Algorithms

A greedy algorithm makes the locally optimal choice at each step, hoping it leads to the globally optimal solution. The power — and the trap — of greedy is that it's not always correct. When it works, it's O(n log n) or O(n). When it doesn't, DP is waiting.

🏆 Locally Optimal → Globally Optimal 🔄 Exchange Argument ⚡ O(n) or O(n log n) 🎯 8 LeetCode problems
Foundation

What Makes Greedy Work?

Greedy works when the problem has the greedy choice property: a locally optimal choice is also globally optimal — you never need to "undo" a decision. This is verified by the exchange argument: assume the optimal solution doesn't make the greedy choice, then show you can swap it for the greedy choice without making things worse.

The Exchange Argument (informal proof): Suppose OPT differs from greedy at some step. OPT made choice X, greedy made choice Y. Show that swapping X for Y in OPT produces a solution that is at least as good. Repeat until OPT equals greedy — proving greedy is optimal.

Classic Example: Coin Change (Greedy FAILS)

With coins [1, 5, 6] and target 10 — greedy picks 6 first, then 1+1+1+1 = 5 coins. But optimal is 5+5 = 2 coins. Greedy fails because coins don't have the "no regret" property. This is why Coin Change needs DP.

Greedy works for Coin Change when: The coin system is "canonical" (like US coins: 1, 5, 10, 25). With standard denominations, each coin is a multiple of smaller ones, ensuring greedy is optimal. With arbitrary denominations, use DP.
Decision Framework

Greedy vs Dynamic Programming

Both greedy and DP break problems into subproblems — the difference is whether you need to reconsider earlier decisions.

Use Greedy when:

A locally optimal choice never needs to be undone. The choice at step i doesn't depend on future choices. Problem has the exchange argument. Examples: activity selection, Huffman encoding, Dijkstra, Kruskal's MST.

Use DP when:

A locally optimal choice might be globally suboptimal. Overlapping subproblems exist. You need to compare multiple options at each step. Examples: coin change with arbitrary denominations, edit distance, 0/1 knapsack.

Pattern 1

Jump Game: Reach / Min Jumps

Jump Game problems ask whether you can reach the end of an array (where each element is your max jump from that position) or how many jumps it takes. Both use greedy — track the furthest reachable position.

Python
# LC #55: Jump Game — can you reach the last index?
def canJump(nums):
    max_reach = 0                    # furthest index reachable so far
    for i, jump in enumerate(nums):
        if i > max_reach:
            return False             # i is unreachable — stuck!
        max_reach = max(max_reach, i + jump)
    return True

# Greedy proof: if we can reach index i, we can reach max_reach.
# We never need to undo — expanding reach is always beneficial.


# LC #45: Jump Game II — minimum number of jumps to reach end
def jump(nums):
    jumps = 0
    curr_end = 0      # end of current jump's "level" (BFS analogy)
    furthest = 0      # furthest we can reach from current level

    for i in range(len(nums) - 1):   # don't need to jump from last index
        furthest = max(furthest, i + nums[i])
        if i == curr_end:            # exhausted current jump's range
            jumps += 1               # must take another jump
            curr_end = furthest      # expand to furthest reachable

    return jumps

# nums = [2,3,1,1,4]
# i=0: furthest=2, curr_end=0 → i==curr_end → jumps=1, curr_end=2
# i=1: furthest=4
# i=2: furthest=4, curr_end=2 → i==curr_end → jumps=2, curr_end=4
# return 2 ✓
Jump Game II visualized as BFS levels: nums = [2, 3, 1, 1, 4] idx = 0 1 2 3 4 Level 0: [0] (start) Level 1: [1, 2] (reach from index 0: 0+2=2, so indices 1 and 2) Level 2: [3, 4] (reach from indices 1,2: max(1+3, 2+1) = 4, so indices 3,4) Minimum jumps = number of levels - 1 = 2 ✓ This is exactly BFS on an implicit graph — greedy picks the optimal "level" boundaries in O(n) vs O(n²) BFS.
Pattern 2

Scheduling: Task Scheduler & Candy

Task Scheduler (LC #621)

Given tasks with a cooldown n between same-task executions, find the minimum time to finish all tasks. The bottleneck is the most frequent task — it determines how many idle slots we need.

Python
def leastInterval(tasks, n):
    from collections import Counter
    freq = Counter(tasks)
    max_freq = max(freq.values())

    # How many tasks share the maximum frequency?
    max_count = sum(1 for f in freq.values() if f == max_freq)

    # Minimum slots needed:
    # (max_freq - 1) "cycles" of (n+1) slots each, plus the last cycle
    # which has exactly max_count tasks (no idle needed at the end)
    min_time = (max_freq - 1) * (n + 1) + max_count

    # But if we have enough tasks to fill all slots, no idle needed
    return max(min_time, len(tasks))

# tasks=["A","A","A","B","B","B"], n=2
# max_freq=3 (A or B), max_count=2
# min_time = (3-1)*(2+1) + 2 = 8
# len(tasks) = 6
# return max(8, 6) = 8
# Layout: A B _ A B _ A B → 8 slots ✓

Candy (LC #135)

Children in a line get ratings. Every child gets ≥1 candy. A child with higher rating than a neighbor must get more candy. Minimum total candies? Two-pass greedy: left-to-right enforces the left neighbor rule, right-to-left enforces the right neighbor rule.

Python
def candy(ratings):
    n = len(ratings)
    candies = [1] * n

    # Left pass: if ratings[i] > ratings[i-1], give one more than left
    for i in range(1, n):
        if ratings[i] > ratings[i - 1]:
            candies[i] = candies[i - 1] + 1

    # Right pass: if ratings[i] > ratings[i+1], give max of current
    # and (right neighbor + 1) — must satisfy BOTH constraints
    for i in range(n - 2, -1, -1):
        if ratings[i] > ratings[i + 1]:
            candies[i] = max(candies[i], candies[i + 1] + 1)

    return sum(candies)

# ratings = [1, 0, 2]
# Left pass:  [1, 1, 2]   (2 > 0: no change since not >, 2 > 0: candies[2]=2)
# Actually: 1>0? No. 2>0? Yes → [1, 1, 2]
# Right pass: [2, 1, 2]   (0 > 2? No. 1 > 0? Yes → max(1, 1+1)=2)
# sum = 5 ✓
Pattern 3

Circular Problems: Gas Station

Gas Station (LC #134): circular route with gas[i] available at station i and cost[i] to travel to the next. Find the starting station (if a solution exists) or return -1.

Key insight 1: If total gas ≥ total cost, a solution always exists (and it's unique).
Key insight 2: If we run out at station j starting from start, then no station between start and j can be the correct starting point either — they would all run out before j. So reset start = j + 1.

Python
def canCompleteCircuit(gas, cost):
    total = 0    # total gas surplus (determines if solution exists)
    tank = 0     # current tank on the candidate route
    start = 0    # candidate starting station

    for i in range(len(gas)):
        diff = gas[i] - cost[i]
        total += diff
        tank += diff

        if tank < 0:
            # Can't reach next station from current start
            # → no station in [start..i] works as start
            start = i + 1
            tank = 0        # reset tank for new candidate

    # If total >= 0, the candidate start is the answer
    return start if total >= 0 else -1

# Why does resetting work?
# If starting at s, we got stuck at j (tank < 0).
# Any start in (s, j]: starting here with fresh tank,
# we'd arrive at j with LESS gas than starting from s
# (because s→start_k contributes positive surplus).
# Since even starting from s failed at j, starting later fails too.
Pattern 4

Greedy Partitioning

Some problems ask you to partition a string or array into maximum parts with a specific property. Greedy here means: extend the current part as far as needed, then cut when possible.

Partition Labels (LC #763)

Partition a string so each letter appears in at most one part. Maximize the number of parts. Greedy: for each position, the current part must extend to the last occurrence of every letter seen so far.

Python
def partitionLabels(s):
    # Precompute last occurrence of each character
    last = {c: i for i, c in enumerate(s)}

    result = []
    start = end = 0

    for i, c in enumerate(s):
        end = max(end, last[c])   # extend partition boundary to include all c's
        if i == end:              # reached the end of current partition
            result.append(end - start + 1)
            start = i + 1        # start new partition

    return result

# s = "ababcbacadefegdehijhklij"
# last = {a:8, b:5, c:7, d:14, e:15, f:11, g:13, h:19, i:22, j:23, k:20, l:21}
# i=0: end=max(0,8)=8
# ...
# i=8: end=8, i==end → part of size 9, start=9
# ...continues → [9, 7, 8]

Two City Scheduling (LC #1029)

Send N people to city A and N people to city B. Person i costs costs[i][0] to go to A, costs[i][1] to go to B. Minimize total cost. Greedy: send everyone to A first, then greedily switch N people to B where the "regret" (B-A cost difference) is smallest.

Python
def twoCitySchedCost(costs):
    # Sort by (cost_B - cost_A): most beneficial to send to B first
    costs.sort(key=lambda x: x[1] - x[0])
    n = len(costs) // 2
    # First n go to B (cheapest to redirect), last n go to A
    return sum(c[1] for c in costs[:n]) + sum(c[0] for c in costs[n:])

LeetCode Problems

#860Lemonade ChangeEasy

Customers pay with $5, $10, or $20. Can you give correct change to everyone? Greedy: when giving change for $20, prefer using $10+$5 over $5+$5+$5 (preserve $5 bills which are more versatile).

Python
def lemonadeChange(bills):
    five = ten = 0
    for b in bills:
        if b == 5: five += 1
        elif b == 10:
            if not five: return False
            five -= 1; ten += 1
        else:  # $20
            if ten and five: ten -= 1; five -= 1   # prefer $10+$5
            elif five >= 3: five -= 3
            else: return False
    return True
⏱ O(n)💾 O(1)Pattern: Greedy — use $10 before $5 for $20 change

#621Task SchedulerMedium

Minimum time to execute all tasks with cooldown n between same tasks.

Python
def leastInterval(tasks, n):
    from collections import Counter
    freq = Counter(tasks)
    max_freq = max(freq.values())
    max_count = sum(1 for f in freq.values() if f == max_freq)
    return max((max_freq - 1) * (n + 1) + max_count, len(tasks))
⏱ O(n)💾 O(1)Pattern: Math formula — max frequency determines idle slots
#135CandyHard

Minimum candies so every child with higher rating than a neighbor gets more candy.

Python
def candy(ratings):
    n = len(ratings)
    candies = [1] * n
    for i in range(1, n):                    # left-to-right pass
        if ratings[i] > ratings[i-1]:
            candies[i] = candies[i-1] + 1
    for i in range(n-2, -1, -1):             # right-to-left pass
        if ratings[i] > ratings[i+1]:
            candies[i] = max(candies[i], candies[i+1] + 1)
    return sum(candies)
⏱ O(n)💾 O(n)Pattern: Two-pass greedy (left constraint, right constraint)
#1029Two City SchedulingMedium

Send N people to city A and N to city B minimizing total cost. Cost[i] = [costA, costB].

Insight: Sort by costB - costA. Sending to A costs costA; "switching" to B costs an extra costB - costA. Sort by this diff ascending — the N people with smallest diff go to B.

Python
def twoCitySchedCost(costs):
    costs.sort(key=lambda x: x[1] - x[0])    # sort by "regret" of sending to A
    n = len(costs) // 2
    return sum(c[1] for c in costs[:n]) + sum(c[0] for c in costs[n:])
⏱ O(n log n)💾 O(1)Pattern: Sort by regret cost
